# Internationalisering (i18n) — Volmer Techniek

## Overzicht

De site ondersteunt meerdere talen via **next-intl 4**. URL-structuur:

- `/nl/` — Nederlands (standaard)
- `/en/` — Engels

Bezoekersvoorkeur wordt opgeslagen in de cookie `NEXT_LOCALE`.

---

## Architectuur

```
/
├── i18n/
│   ├── routing.ts      — Locale-definitie en standaard locale
│   ├── navigation.ts   — Typed Link, useRouter, usePathname
│   └── request.ts      — Server-side message loader
├── messages/
│   ├── nl.json         — Nederlandse vertalingen
│   └── en.json         — Engelse vertalingen
├── middleware.ts        — Locale-detectie, cookie, redirect
├── app/
│   ├── layout.tsx      — Root layout (html lang attr)
│   └── [locale]/
│       ├── layout.tsx  — NextIntlClientProvider + metadata
│       └── page.tsx    — Homepagina
└── types/
    └── next-intl.d.ts  — TypeScript type augmentation
```

### Hoe locale-detectie werkt

1. Bezoeker komt op `/` — middleware detecteert `Accept-Language` header
2. Als `NEXT_LOCALE` cookie aanwezig → gebruik die waarde (geen redirect)
3. Anders → redirect naar `/nl` of `/en` op basis van browser-taal
4. Bij taalkeuze via de switcher: cookie `NEXT_LOCALE` wordt gezet voor vervolgbezoeken

## Een nieuwe taal toevoegen (bijv. Duits)

### Stap 1 — Voeg de locale toe aan routing

Bestand: [i18n/routing.ts](../i18n/routing.ts)

```ts
export const routing = defineRouting({
  locales: ["nl", "en", "de"],   // ← voeg "de" toe
  defaultLocale: "nl",
});
```

### Stap 2 — Maak het vertaalbestand aan

Kopieer `messages/en.json` naar `messages/de.json` en vertaal alle waarden.

Structuur: identiek aan `nl.json` en `en.json`. Verander alleen de waarden, nooit de keys.

### Stap 3 — Voeg de taal toe aan de request config

Bestand: [i18n/request.ts](../i18n/request.ts)

Geen aanpassing nodig — de config laadt automatisch `messages/${locale}.json`.

### Stap 4 — Voeg metadata toe voor SEO

Bestand: [app/[locale]/layout.tsx](../app/[locale]/layout.tsx)

Voeg de nieuwe taal toe aan de `languages` in `generateMetadata`:

```ts
languages: {
  nl: `${siteUrl}/nl`,
  en: `${siteUrl}/en`,
  de: `${siteUrl}/de`,   // ← voeg toe
  "x-default": `${siteUrl}/nl`,
},
```

### Stap 5 — Voeg de locale toe aan het type-systeem

Bestand: [types/next-intl.d.ts](../types/next-intl.d.ts)

Geen aanpassing nodig — types worden afgeleid van `nl.json`.

### Stap 6 — Test

```bash
npm run dev
# Ga naar http://localhost:3000/de
```

## Vertaalbestanden — structuur

Alle vertalingen staan in JSON-bestanden onder `messages/`. De structuur is genest per sectie:

```json
{
  "metadata": { ... },        // SEO: title, description, keywords
  "common": { "skip": "..." }, // Toegankelijkheidsstrings
  "header": { ... },          // Navigatie, CTA, mobile menu
  "hero": { ... },            // Hero-sectie
  "services": { ... },        // Dienstensectie incl. items[]
  "workflow": { ... },        // Werkwijze incl. steps[]
  "about": { ... },           // Over ons incl. facts[]
  "contact": { ... },         // Contactgegevens, formulier, errors
  "footer": { ... },          // Footer, kolommen, links
  "languageSwitcher": { ... } // Aria-label taalkiezer
}
```

### Arrays

Arrays (services.items, workflow.steps, about.facts, contact.form.typeOptions) zitten in het JSON-bestand als gewone JSON-arrays. In componenten worden ze opgehaald via `t.raw('items')`.

### Wat NIET vertaald wordt

Per afspraak uit `content/index.ts` zijn dit geen vertalingen:
- Bedrijfsnaam "Volmer Techniek" — eigennaam
- NAP-data (telefoon, e-mail, adres) — per taal gelijk, wel in messages gezet voor consistentie
- Decoratieve symbolen (`◆ ■ —`)
- Numerieke labels (`01`, `02`)
- Ankertekst section-IDs (`#diensten`, `#werkwijze`) — technische identifiers

## Ontwikkelnotities

- **Server components** gebruiken `getTranslations('namespace')` (async)
- **Client components** gebruiken `useTranslations('namespace')` (synchronous hook)
- **Arrays** ophalen via `t.raw('key') as TypeName[]`
- **TypeScript** type-safety via `types/next-intl.d.ts` (afgeleid van `nl.json`)
- **Cookie**: `NEXT_LOCALE` — 1 jaar geldig, SameSite=Lax
- **Middleware**: verwerkt alle requests behalve `_next`, `_vercel` en statische bestanden
